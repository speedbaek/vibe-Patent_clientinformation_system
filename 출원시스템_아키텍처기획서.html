🔐 키 관리 구조

  • 마스터 키 (KEK): AWS KMS 또는 사내 HSM에서 생성·보관
  • 데이터 키 (DEK): 마스터 키로 암호화하여 DB에 저장
  • 키 로테이션: 분기별 (90일) 자동 교체, 이전 키로 복호화 후 신규 키로 재암호화
  • 키 백업: 마스터 키는 KMS 자동 백업, DEK는 DB 백업에 포함
  • 키 폐기: 데이터 삭제 후 90일 뒤 관련 DEK 영구 삭제

📜 TLS 인증서 관리

  • 인증서 발급: Let's Encrypt 또는 유료 CA (DigiCert 등)
  • 갱신 주기: 자동 갱신 (만료 30일 전)
  • TLS 버전: 최소 TLS 1.2, 권장 TLS 1.3
  • 취약 암호 스위트: RC4, DES, 3DES 비활성화
  • HSTS: max-age=31536000, includeSubDomains
💎
암호화 적용 범위 요약 전송 구간: TLS 1.2+ (전체) · 저장소: AES-256-CBC (주민번호, 법인번호, 서명, 도장) · 파일: S3 SSE-KMS 또는 AES-256 자체 암호화

고객 UX 핵심 포인트
"쉽고 안심되는" 경험이 제출 완료율을 높인다
✅ 이렇게 해야 한다
• 이메일에 링크 하나만 — 로그인/회원가입 없음
• 모바일 최적화 (손가락으로 서명, 카메라로 서류 촬영)
• 진행률 표시 (Step 1/5 → 시각적 안도감)
• "지금 못 내도 나중에 이메일로 추가 제출 가능" 명확 안내
• 자동 저장 (임시저장) → 실수로 창 닫혀도 복구
• 입력 오류 즉시 인라인 안내 (빨간 밑줄 + 메시지)
• 개인정보 수집 목적 한 줄 요약 + 전문 링크
• 제출 완료 후 접수번호 + 이메일로 확인서 발송
🚫 이렇게 하면 안 된다
• 회원가입 요구 → 고객 이탈율 급등
• 모든 서류를 한 번에 다 받으려는 설계 → 제출 포기
• 주민번호 입력창에 아무 안내 없음 → 불안감
• 에러 메시지가 "오류가 발생했습니다"처럼 불분명
• 링크 만료 후 그냥 404 → 재발급 안내 없음
• PC 전용 설계 (모바일에서 서명 불가)
• 저장 없이 새로고침하면 처음부터 다시 → 이탈
• 서명·도장을 FAX로 받는 기존 방식 병행 없음
